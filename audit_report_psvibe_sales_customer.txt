4. GOOGLE SHEETS INTEGRATION
================================================================================

4.1 SHEET INITIALIZATION (CRITICAL BLOCKING)

  All 9 worksheets are opened at module import time (lines ~62-70 of __init__.py):
    sales_sh, setting_sh, member_sh, stock_sh, stock_in_sh, topup_sh, inv_sh
  Plus on-demand worksheets: att_sh, booking_sh, salary_adv_sh, game_lib_sh,
    console_games_sh, and finance sheets.

  This means:
    - ANY sheets API failure at startup kills the entire bot
    - Bot cannot "degrade gracefully" if one sheet is temporarily unavailable
    - Bot restart loops if Sheets API is down

  Severity: HIGH

4.2 RETRY WRAPPER

  GOOD: gspread Worksheet methods are monkey-patched with _sheets_retry decorator
    - Retries on 429, 500, 503 with exponential backoff (1s/2s/4s)
    - Also retries on ConnectionError, TimeoutError, OSError
    - Max 3 retries, then propagate

  Issue: Only gspread.Worksheet methods are wrapped. Direct calls to
    gspread.Spreadsheet or gspread.Client are NOT wrapped (except
    add_worksheet which is explicitly wrapped).

4.3 CACHING STRATEGY

  Config:  5 min TTL (base_rate, console_multipliers, bonus_table, etc.)
  Members: 3 min TTL (Card_Wallet rows)
  Booking: 30 sec TTL (active session overlay)
  Games:   10 min TTL (Game_Library rows)
  Console Games: 5 min TTL
  Promotions: 2 min TTL

  Issue: fetch_balance_mins() bypasses cache (calls get_all_values directly)
    — correct for wallet balance but adds API calls per interaction

  Issue: check_disc_session_conflict() calls get_all_values() directly
    — could be cached in the 30s booking cache

4.4 API CALL ANALYSIS PER TYPICAL SALES FLOW

  A single Daily Sale (new sale → select member → select console → enter mins
  → discount → confirm) triggers approximately:
    - 6-10 sheets API calls (member lookup, console lookup, balance check,
      rate check, write sale, update wallet)
    - Plus API server calls for receipt push

  This is acceptable for a manual-use internal tool but could be optimized
  by batching writes.

5. ERROR HANDLING REVIEW
================================================================================

5.1 GLOBAL ERROR HANDLER

  handlers.py line 11538:
    async def error_handler(update, context):
      - Logs NetworkError/TimedOut as WARNING (auto-retry)
      - Logs Conflict as WARNING (auto-resolve)
      - Logs everything else as ERROR with exc_info

  GOOD: Prevents bot crashes from unhandled exceptions
  ISSUE: No user notification on failure. A staff member submits a sale,
    it fails silently (just logged), and no retry/notification happens.

5.2 TRY/EXCEPT PATTERNS

  handlers.py: ~200+ try/except blocks
    - Most catch generic Exception
    - Many silently suppress (except: pass or except Exception: pass)
    - Some log warnings but continue as if nothing happened

  Customer Bot: Better structured
    - Specific HTTP error handling (HTTPError, URLError, TimeoutError)
    - API client has built-in retry for transient errors
    - Gemini AI failures fall back gracefully

5.3 SILENT FAILURES (Examples)

  - end_booking(): "except Exception: pass; return False"
  - cancel_booking(): Same pattern
  - fetch_console_status(): "except Exception: pass" at end
  - fetch_games(): "except Exception: return []"

  Severity: MEDIUM — Data integrity risk if writes fail silently

5.4 MISSING ERROR RECOVERY

  - No retry queue for failed writes
  - No dead-letter queue
  - No idempotency keys on API calls (double-write risk on retry)

6. TELEGRAM API & CONVERSATIONHANDLER
================================================================================

6.1 CONVERSATION HANDLER COMPLEXITY

  States: 177 (range(177))
  Each state has 1 MessageHandler(filters.TEXT & ~filters.COMMAND, handler)
  Total handler entries in states dict: ~100+
  Fallback command handlers: 35+

  Severity: HIGH — This is the largest ConversationHandler I've ever seen.
  The Python Telegram Bot library is not designed for state machines this large.
  Each incoming message must iterate through ALL states looking for a match.

6.2 GROUP HANDLER ORDER

  Group -999: Auth middleware (TypeHandler)
  Group -1:   Custom extend reply handler
  Group 0:    ConversationHandler (implicit)
  Group 10:   Cancel note input handler

  Plus global CallbackQuery handlers (no group, added after conv)

  This ordering means the auth middleware blocks everything before it
  reaches the conversation handler — GOOD for security.

6.3 CUSTOMER BOT APPROACH

  The customer bot (monolithic) has a simpler handler setup:
    - One main ConversationHandler with ~10 states
    - All free-text messages → Gemini AI response
    - Bot commands (/start, /menu, etc.) as entry points and fallbacks

  This is architecturally cleaner but the monolithic structure is problematic.

6.4 TELEGRAM API CALL FREQUENCY

  Both bots use long-polling (getUpdates) — visible in logs at ~10s intervals
  This is the standard python-telegram-bot behavior and acceptable.

  No webhook usage; polling is used exclusively.
