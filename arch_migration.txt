=== 4. STRUCTURED ANALYSIS ===

### CURRENT STATE
- Hybrid architecture: READ via API → MySQL, WRITE still mostly direct GSheets
- READ operations: Already migrated to API Client → API Server → MySQL
- WRITE operations: Mixture of API calls + direct GSheets append_row()
- Direct sheets writes: ~10 calls across ~5 handler files
- Direct GSheets READ access: Also still possible via gspread in __init__.py

### PROS of Full Migration
1. Data Integrity — MySQL ACID transactions eliminate sheet race conditions
2. Latency — Localhost API (~0.001-0.01s) vs GSheets internet (~0.5-2s)
3. Rate Limits — MySQL has no quota; GSheets 60 req/min/user, 300 req/min/project
4. Offline Resilience — MySQL local; no internet needed for core ops
5. Audit Trail — MySQL binlog + structured logging
6. Sync Simplification — 1-directional MySQL → GSheets only
7. Concurrency — MySQL handles 64+ concurrent writes (vs GSheets sequential)
8. Consistency — Single source of truth (MySQL), sheets are just mirrors

### CONS of Migration
1. Write Endpoint Gap — API server lacks write endpoints for finance/stock/salary
2. API Client Brittleness — No retry, no timeout handling, no error differentiation
3. Effort — ~45 handler files to audit, ~5 with direct writes to convert
4. Sync Critical Path — MySQL → GSheets sync must be reliable
5. Dual-Write Risk — Transition period complexity
6. Monitor Overhead — Need to monitor sync health + API health
