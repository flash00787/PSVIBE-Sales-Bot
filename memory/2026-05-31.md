## 🔴 Bug Fix Round 3.5 — Garbled Burmese Unicode Escapes (16:20-16:34 UTC)

### Bug: Wrong Burmese text in TU/NM payment amount prompts (Image 3 from Boss)
- **Root cause:** My `fix_confirm_flow.py` used wrong `\u` unicode escape sequences in `members.py`:
  - `\u1019\u1019\u102d\u1014\u103a` = **မမိန္** (wrong) → should be `\u1015\u1019\u102b\u100f` = **ပမာဏ** (pa-ma-nya = "amount")
  - `\u100a\u1037\u1014\u1031\u1000\u103a` = **ညေနက** (wrong) → should be `\u1011\u100a\u1039\u1037\u1015\u102b\u1038` = **ထည့်ပါ** (hter-par = "enter/put in")
- Also fixed: `\u101b\u102d\u102f\u1000\u103a\u1015\u102b\u1038` → removed trailing `\u1038` (း) — "ရိုက်ပါး" → "ရိုက်ပါ"
- Bug appeared in **BOTH** TU (`step_tu_kpay`) and NM (`step_nm_kpay`) amount prompts (2 occurrences each = 4 total fixes)

### Fix Applied:
- `/tmp/fix_unicode_v3.py` — Replaced wrong unicode escape sequences with correct ones
- `/tmp/fix_unicode_v4.py` — Corrected `\u1013` (ဓ) → `\u100f` (ဏ) and removed trailing `\u1038`
- Service restarted: `psvibe-sale-bot` ✅ active

### Lesson Learned: \u escapes in fix scripts are fragile
- `\u100f` = ဏ (nya), NOT `\u1013` = ဓ (da) — subtle but critical difference
- Best approach: write direct Burmese string literals instead of constructing `\u` escapes
- Always verify: `python3 -c "print('\\u1015\\u1019\\u102b\\u100f')"` — should display readable Burmese
