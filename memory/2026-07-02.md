### Booking vs Session Separation — Comprehensive Audit

**Spawned Pro sub-agent** → full audit report at `memory/booking-session-separation-audit.md`

### Core Problem
`console_booking` table mixes TWO conceptually different things:
| | **Booking** | **Session** |
|---|---|---|
| Meaning | Future reservation (ကြိုတင်စာရင်း) | Actual play time (ကစားချိန်) |
| Status flow | pending → confirmed → rejected/cancelled | checked_in → Active → Done |
| Time | Planned ("3PM လာမယ်") | Actual ("ခု 3:15 စဆော့") |
| Worst bug | `start_time` gets OVERWRITTEN during checkin → original booking time LOST |

### Status Mixing (10 statuses, mixed booking + session)
`pending`, `confirmed`, `rejected`, `cancelled`, `pending_check_in`, `checked_in`, `Active`, `Done`, `Waiting`, `Notified`

## Session 2: Booking vs Session — Full Implementation Plan (~10:00 AM MMT)

### Boss Decision
1. ✅ `start_time` overwrite during checkin is **CORRECT** — keep as-is
2. ✅ FULL separation: Booking table + Session table, completely split
3. ✅ Need detailed step-by-step executable plan

### Work Done
Spawned Pro sub-agent → `memory/booking-session-separation-plan.md` (890 lines)

### Plan Highlights

**Schema — 2 New Tables:**
| | `console_bookings` | `console_sessions` |
|---|---|---|
| Time columns | `planned_start` / `planned_end` | `actual_start` / `actual_end` |
| Statuses | pending, confirmed, rejected, cancelled, pending_check_in, waiting, notified | checked_in, active, done |
| Link | — | `booking_id` FK → console_bookings.id (ON DELETE SET NULL) |
| Walk-in | — | `booking_id = NULL` |

**Migration:**
- Step A: CREATE 2 new tables
- Step B: pending/confirmed/rejected/cancelled → console_bookings
- Step C: checked_in/Active/Done → console_sessions (with booking_id link)
- Step D: Create booking rows for completed sessions
- Step E: Verify counts
- Dependent tables: food_cart, stock_hold, customer_feedback, sales_daily → add `session_id` column, dual-column transition

**Phase Plan:**
| Phase | Scope | Hours | Risk |
|-------|-------|-------|------|
| 1. Schema | DDL, migration SQL, backup | 4-6 | 🟢 LOW |
| 2. API | 5 route files, ~150+ SQL queries | 25-30 | 🔴 HIGH |
| 3. Bots | Sale Bot 4 files, Customer Bot 2 files | 12-15 | 🔴 HIGH |
| 4. Dashboard | ~10 Vue views | 10-12 | 🟡 MEDIUM |
| 5. Test | 8 integration test scenarios | 8-10 | 🟡 MEDIUM |
| 6. Cleanup | Legacy table, docs | 2-3 | 🟢 LOW |
| **TOTAL** | **~33 files, ~6,800 lines** | **~60-75 hrs** | |

**Key Decisions:**
- `booking_id` FK ON DELETE SET NULL — session survives booking deletion
- Session statuses lowercase (active/done) vs Booking original case
- Food cart dual-column transition strategy
- `console_bookings.console_id` NULLABLE for unassigned bookings
- 8 integration test scenarios defined (booking flow, walk-in, cancel, move/swap, food cart, reconciliation, dashboard, console swap)

1. `food_cart.booking_id` = VARCHAR(50) not INT
2. `stock_hold.booking_id` = VARCHAR(64) not INT
3. `sales_daily` has NO booking_id column (reconciliation = console_id+time+mins)
4. `branch_id` column missed in v1 plan
5. Status case: `Active` (capital) vs `active` (lowercase) — case mismatch risk
6. `Waiting`/`Notified` uppercase in legacy → migration must lowercase
7. Timer module uses `booking_id` as dict key → must change to `session_id`
8. `console_status.py` & `session_timer.py` use SEPARATE MySQL connections
9. `session_timer.py` doesn't use `mysql_db.py` — must update BOTH
10. Checkin endpoint does NOT start timer (Sale Bot's `_remind_loop` does)
