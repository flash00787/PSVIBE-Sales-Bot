## FIX (3) — VPN Admin Page Speed Optimization: Metrics Cache + Pre-fetch

- `[fixed [high]]` **Outline Admin Page 20-40s Load Time → <0.5s (2026-07-09)**  
  *vpn, outline, prometheus, metrics, speed, cache* | `id:a2ebdb18`
  > **Root Cause:** Every admin page load called `get_key_metrics()` per active Outline key (4 keys), each hitting Prometheus API. TH server keys required SSH subprocess (15s timeout) → total load 20-40s. Cloudflare timeout at 100s then crashes the page.
  > **Fix Chain (3 layers):**
  > - **Layer 1 (12:00 UTC):** `get_key_metrics()` timeout reduced from 10-15s → **3s**. Metrics loop in render_keys() + agent render + admin page got **5s global timeout** — once exceeded, remaining metrics skip. Page load improved from 20-40s → ≤5s.
  > - **Layer 2 (13:38 UTC):** Global `_metrics_cache` dict added with 30s TTL. Subsequent page loads within 30s return cached data instantly. Page load ≤0.5s on second hit.
  > - **Layer 3 (14:12 UTC):** Background pre-fetch thread (`_prefetch_loop()`) runs every 25s, warming the metrics cache automatically. First load is now also instant — cache always fresh.
  > **Files:** `/opt/outline-web/server.py` — `get_key_metrics()` timeout param, `_metrics_cache` dict, `_prefetch_loop()` thread

## LESSON (5)

- `[verified [high]]` **Prometheus metrics must be cached for admin page performance**  
  *lesson, vpn, prometheus, cache* | `id:a2ebdb22`
  > #143 — Fetching Prometheus per-key metrics on every page load is prohibitively slow. Key metrics change slowly (bytes used grows gradually). A 30s TTL cache is safe and dramatically improves UX.

- `[verified [high]]` **Background pre-fetch > lazy fetch for admin dashboards**  
  *lesson, vpn, pre-fetch, performance* | `id:a2ebdb23`
  > #144 — Even with caching, the first page load after cache expiry still blocks on Prometheus queries. A background thread that pre-warms the cache at a shorter interval (25s) than the TTL (30s) ensures cache is always fresh — every page load is instant.

- `[verified [high]]` **Cloudflare proxied Caddy + synchronous subprocess = crash**  
  *lesson, vpn, cloudflare, crash, background* | `id:a2ebdb24`
  > #145 — Cloudflare's proxy has a 100s timeout for HTTP responses. Synchronous Outline API DELETE calls (especially TH server via SSH) can hang indefinitely. Always move Outline API calls to background threads in the HTTPServer architecture. The page response returns immediately; the key deletion happens asynchronously.

- `[verified [high]]` **Google revokes API keys detected as leaked**  
  *lesson, google, api-key, leak* | `id:a2ebdb25`
  > #146 — Google actively monitors for leaked API keys (committed to public repos, shared in messages, etc.). Once flagged, the key returns HTTP 403 "Your API key was reported as leaked" and must be replaced. Affects: Gemini embeddings, image analysis, any Google API call. Keep multiple working keys as backup.

- `[verified [medium]]` **GEMINI_API_KEY and GOOGLE_API_KEY may be separate variables**  
  *lesson, google, env, configuration* | `id:a2ebdb26`
  > #147 — In the OpenClaw gateway, GEMINI_API_KEY is used for Gemini-specific features (embeddings, vision). GOOGLE_API_KEY may point to a different key. Always test both: `curl -X POST -H "Content-Type: application/json" -H "x-goog-api-key: $KEY" "https://generativelanguage.googleapis.com/v1beta/models?key=$KEY"` returns 200 or 403.

- `[verified [medium]]` **Coco docker-compose uses same env vars as gateway**  
  *lesson, coco, docker, env* | `id:a2ebdb27`
  > #148 — `oc-coco` container at `/opt/coco/docker-compose.yml` uses GEMINI_API_KEY and GOOGLE_API_KEY environment variables. When updating leaked keys in the OpenClaw gateway, must also update the coco docker-compose and recreate the container. Check `CONTAINER_ID=$(docker ps -q -f name=coco)` and `docker compose up -d`.
