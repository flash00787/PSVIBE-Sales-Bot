## Cancelled Bookings Timeline — 2-Hour Filter (06:20-06:48 UTC / 12:50-13:18 MMT)

### Problem
Boss requested cancelled bookings to only show for 2 hours on the booking timeline. Previously they appeared indefinitely.

### Root Cause Analysis
1. **No `cancelled_at` column** — DB had no field tracking when a booking was cancelled. The filter was using `end_time` (scheduled slot end), which was wrong.
2. **API returns snake_case** — `cancelled_at` not `cancelledAt`, causing JS filter to miss the field.
3. **Timezone mismatch** — `new Date("2026-06-27T04:15:00")` in JS interprets as local time (MMT), not UTC. MySQL stores in UTC. Added `+ 'Z'` suffix to force UTC parsing.

### What was done

#### 1. DB Migration
- **Added** `cancelled_at DATETIME` column to `console_booking` table
- **Backfilled** 175 existing cancelled bookings with `cancelled_at = created_at` (best estimate)
- **Manual fix**: ID 1035 corrected to 10:45 AM MMT per Boss instruction

#### 2. API Updates (`/root/psvibe_api_server/app.py`)
- **3 cancel endpoints** updated to set `cancelled_at = NOW()`:
  - `PATCH /api/bookings/{id}/status` (status transition to cancelled)
  - Cancel booking endpoint
  - Auto-cancel expired pending
- **Server-side filter**: `cancelled_at >= NOW() - INTERVAL 2 HOUR` added to GET bookings query (only when status=cancelled)
- **`cancelled_at`** added to SELECT query

#### 3. Frontend Updates (`/root/psvibe-dashboard/src/api/timeline.ts`)
- Filter uses `cancelled_at` (snake_case) with fallback to `cancelledAt` (camelCase)
- **UTC parsing fix**: `new Date(cancelledAt.replace(' ', 'T') + 'Z')` — 'Z' suffix forces UTC regardless of browser timezone
- Fallback to `end_time` for old bookings without `cancelled_at`
- Safety: if no timestamp at all → show booking (don't hide)

#### 4. Architecture: Double-Layer Defense
- **Layer 1 (Server)**: MySQL `NOW() - INTERVAL 2 HOUR` — authoritative filter
- **Layer 2 (Frontend)**: JS filter with UTC-safe `+Z` parsing — defense-in-depth

### Files Modified

## Client-Side API Format Bugs — Full Fix Suite (14:00-14:30 UTC / 20:30-21:00 MMT)

### unwrap_response() + Nested "data" Key Bug

**Bug:** `customer_bot/api.py` `_api_get()` wraps responses with `unwrap_response()` which strips `.data` wrapper. But consuming code in `_get_linked_member_id()` and `_get_linked_phone()` tried:
```python
bookings = data.get("data", {}).get("bookings", [])  # ❌ always empty
```
`unwrap_response()` already returned `{"bookings": [...], "total": 25}` — no nested `"data"` key.

**Fix:**
```python
bookings = data.get("bookings", [])  # ✅ direct access
```

### _urllib.request Bug

**Bug:** `dashboard_routes.py:import urllib.request as _urllib` — `_urllib` IS the `urllib.request` module. Code called:
```python
_urllib.request.urlopen(req)  # ❌ = urllib.request.request.urlopen (doesn't exist)
```
**Fix:** `_urllib.urlopen(req)` — direct module-level call.

### Cancelled Booking Skip — Linkage Data Loss

**Bug:** `_get_linked_member_id` skipped cancelled bookings entirely:
```python
if b.get("status") == "cancelled": continue  # ❌ throws away member_id+tg link
```
**Fix:** Remove the filter — cancelled bookings still hold valid `member_id` → `telegram_chat_id` mappings.

### Files Modified
- `/root/psvibe-sales-bot/customer_bot/api.py` — 3 fixes
- `/root/psvibe_api_server/dashboard_routes.py` — _urllib fix

### New Lessons
- **#44: unwrap_response() changes response structure** — Functions consuming API responses must NOT access `.data` again after unwrap. Use `data.get("bookings", [])` not `data.get("data", {}).get("bookings", [])`.
- **#45: `import X as Y` aliasing** — `import urllib.request as _urllib` means `_urllib` IS the module. Call `_urllib.urlopen()` not `_urllib.request.urlopen()`.
