**Issue 1 — Missing Endpoints (2 removed during cleanup)**

|----------|-----------------|--------|

Both were simple `logger.info()` endpoints — no GSheet or MySQL needed. Restored to `app.py` with same behavior (accept payload, log to logger, return `{"logged": True}`/`{"tracked": True}`).

**Issue 2 — Route Order Bug: `/api/bookings/search` 404**
- **FastAPI route matching is order-dependent**: `@app.get("/api/bookings/{booking_id}")` was registered BEFORE `@app.get("/api/bookings/search")` causing "search" to be captured as a `{booking_id}` parameter
- **Fix**: Swapped route registration order in `patch_routes.py` — search now comes before `{booking_id}`
- **Test**: `GET /api/bookings/search?date=2026-06-01` → 200 OK (1 booking returned)

### Final Verification
- `POST /api/sheets/log` → 200 ✅
- `POST /api/bot-users/track` → 200 ✅
- `GET /api/bookings/search?date=2026-06-01` → 200 ✅
- `GET /api/bookings/search` → 200 (26 bookings) ✅
- All 3 services active (API, Sales Bot, Customer Bot) ✅

### Lesson: FastAPI Route Order Matters
When paths overlap semantically (`/api/bookings/search` vs `/api/bookings/{booking_id}`), the **specific path MUST be registered BEFORE the parameterized path**. FastAPI matches routes in registration order — `{booking_id}` captures any single path segment including "search".

### Endpoint Recovery Pattern
For future reference: when an endpoint is missing after migration:
1. Check backup files (`.bak.migration_*`) for original code
2. Check if it's a fire-and-forget logger (just needs `logger.info()` + `return ok({})`)
3. Check route order if path overlaps with parameterized routes
4. Always smoke-test ALL endpoints after any migration batch
