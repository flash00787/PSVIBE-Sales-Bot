## 🐛 Bug Fix: Session End — Game Minutes Missing When Food Orders Present

### Root Cause
`cmd_session_food_order()` in `sales.py` sets `session_food_order: True` and `is_food_sale: True` in context. When staff later ends the session via `step_end_confirm()` in `console.py`, `launch_session_sale()` uses `context.user_data.update()` which doesn't clear old flags. The stale `session_food_order=True` flag causes `step_food_menu` Done button to route to `_save_food_cart()` instead of `prompt_confirm()` — so the sales voucher never shows game minutes, only food cart items.

### Fix Applied (2 locations)
1. **`/root/psvibe-sales-bot/bot/handlers/sales.py`** — `launch_session_sale()`: Clear `session_food_order`, `is_food_sale`, `last_food` before `context.user_data.update()`
2. **`/root/psvibe-sales-bot/bot/handlers/console.py`** — `step_end_confirm()`: Clear same flags before calling `launch_session_sale` (belt-and-suspenders)

- `psvibe-sale-bot` (systemctl)

## 🧮 Balance Sheet & Cash Flow Fixes

### Balance Sheet Imbalance (102.65M gap)

**Root Cause:** Settled advance recovery inject (102.5M) is in KBZ Bank cash balance but NOT reflected in retained earnings. The retained earnings formula only uses `sales_daily net + wallet_consumed - opex - cogs - depreciation`, which ignores cash_movements inject records.

**Fix:** Added settled advance recovery to retained earnings calculation:

```python
# In dashboard_routes.py and patch_routes.py:
_adv_recovered = float(_mqo("SELECT COALESCE(SUM(amount),0) FROM cash_movements WHERE movement_type='inject' AND account='KBZ Bank' AND note LIKE 'Advance settled%%'")["t"] or 0)
retained += _adv_recovered
```

Also removed the 100,000 Ks auto-balancer cap → allows any size gap to be auto-balanced.

**Result:**
- Retained Earnings: -6,656,145 → +95,996,167 Ks
- Balance Diff: 102,652,312 → 0 ✅

### Cash Flow Reconciliation

**Root Cause:** `net_change = net_op + net_inv` didn't include internal account transfers (transfer_in/transfer_out). The closing balance (`total_bal` from per-account loop) WAS correct, but displayed net_change didn't reconcile.

**Fix:** Changed `net_change = round(total_bal - opener, 0)` — derived directly from closing and opening balances instead of summing inflows/outflows.

**Result:** Opening + Net Change = Closing ✅
