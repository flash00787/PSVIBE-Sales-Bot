## 6. ­čÉŤ C05 ÔÇö Booking Creation Overwrites Active Console Status ÔÜá´ŞĆ (NOT YET FIXED)

### Boss Report
C-05 showing as Free (actually "Reserved" in DB) while session was still Active.

### Root Cause Found ÔÇö `POST /api/bookings` at app.py line ~2845
When a new CONFIRMED booking starts within 60 minutes, the endpoint **directly overwrites** `console_status` to `Reserved` WITHOUT checking if there's an Active session:

```python
# app.py line ~2845 ÔÇö THE BUG
if start_dt <= now_mmt() + timedelta(minutes=60):
    cur.execute(
        "UPDATE console_status SET status='Reserved', current_member=%s, start_time=NULL WHERE console_id=%s",
        (bk_member_id or customer_name, console_id)
    )
```

### Timeline
| Time (MMT) | Event |
|------------|-------|
| 12:38 | C-05 Free Ôćĺ Reserved (booking #961 for 13:45) |
| 13:45 | C-05 Reserved Ôćĺ Active (R gyi, 120 min, ends 15:45) |
| **15:18** | **Booking #975 created** (Sai Aung Hein Ko, 16:00-18:00) Ôćĺ **Active overwritten to Reserved** |
| 15:35 (now) | C-05 still Reserved, booking #961 still Active in DB ÔÇö status drift |

### Fix Plan
Replace direct `UPDATE console_status SET status='Reserved'` with `_sync_console_status(console_id)` which properly checks for Active bookings first:
1. Active booking found Ôćĺ keep Active Ôťů
2. No Active + confirmed within 60 min Ôćĺ Reserved Ôťů
3. Neither Ôćĺ Free Ôťů

## 8. Booking #974 Disappearance ÔÇö Root Cause & Fixes Ôťů (17:00 MMT)

### Investigation Result
Staff Booking flow on C-03 Ôćĺ `step_book_checkin_bind` called **`POST /api/update_booking`** (endpoint doesn't exist Ôćĺ HTTP 405). But `_api_call_async` catches HTTPError and returns dict instead of raising Ôćĺ code thought checkin succeeded (logged success) but booking was NOT updated to Active. Later `_sync_console_status` ran, found no Active booking Ôćĺ set C-03 Free Ôćĺ booking cancelled.

### Fix 1: Dead API Endpoint (HTTP 405)
- **File:** `/root/psvibe-sales-bot/bot/handlers/booking.py` line 1213
- **Before:** `_psvibe_post_async("update_booking", {"id": bk_id, "status": "Active", "console_id": cid})`
- **After:** `_psvibe_patch_async(f"bookings/{bk_id}/status", {"status": "Active", "consoleId": cid})`
- Service: `psvibe-sale-bot` restarted Ôćĺ PID 2294683

### Fix 2: _sync_console_status ÔÇö NOW() (UTC) Ôćĺ now_mmt()
- **File:** `/root/psvibe_api_server/app.py` lines 96-158
- **Before:** SQL used `NOW()` (MySQL server UTC) to compare against `start_time` (stored as MMT) Ôćĺ ~6.5hr mismatch
- **After:** Generate `_now_mmt_str` and `_now_plus_60` via Python `now_mmt()`, use parameterized `%s`
- Also fixed `DATE_ADD(NOW(), INTERVAL 60 MINUTE)` and `end_time >= NOW()` clauses

### Fix 3: Booking Ôćĺ Active: start_time NOW() Ôćĺ MMT
- **File:** `/root/psvibe_api_server/app.py` line 1623
- **Before:** `start_time=NOW()` in `PATCH /api/bookings/{id}/status` when statusÔćĺActive
- **After:** `start_time=%s` with `now_mmt().strftime('%Y-%m-%d %H:%M:%S')`
- Service: `psvibe-api` restarted Ôćĺ PID 2294664

### New Lessons
- **#29: HTTP error handling asymmetry** ÔÇö `_api_call_async` returns error dict for 4xx (no exception) but `_psvibe_post_async` catches Exception Ôćĺ callers never know API failed. Check if response has `status_code` key.
- **#30: MySQL NOW() is UTC** ÔÇö All time comparisons in SQL must use app-level MMT time passed as parameter. Direct `NOW()` in queries against MMT-stored datetimes will ALWAYS be wrong.
- **#31: Minified Vue SPA fixes must cover BOTH save logic AND template render** ÔÇö Removing a `v()` condition from save function (`I()`) is not enough; the template ternary `v() ? ... : y("",!0)` also gates the INPUT fields. Miss this = fields invisible, save sends nothing.

### Fix 4-5: Dashboard Edit Booking ÔÇö Template v() Removal
- **File:** `/root/psvibe_api_server/dashboard-dist/assets/BookingsManagement-BbqxMmPA.js`
- **Fix 4:** `,if(o.value.booking_date...` Ôćĺ `,o.value.booking_date...` (stray `if` after removing `v()){` wrapper)
- **Fix 5:** `v()?(d(),i(T,{key:0},[...time/duration fields...],64)):y("",!0)` Ôćĺ `(d(),i(T,{key:0},[...time/duration fields...],64))` (template always renders fields)
