## 6. 🐛 C05 — Booking Creation Overwrites Active Console Status ⚠️ (NOT YET FIXED)

### Boss Report
C-05 showing as Free (actually "Reserved" in DB) while session was still Active.

### Root Cause Found — `POST /api/bookings` at app.py line ~2845
When a new CONFIRMED booking starts within 60 minutes, the endpoint **directly overwrites** `console_status` to `Reserved` WITHOUT checking if there's an Active session:

```python
# app.py line ~2845 — THE BUG
if start_dt <= now_mmt() + timedelta(minutes=60):
    cur.execute(
        "UPDATE console_status SET status='Reserved', current_member=%s, start_time=NULL WHERE console_id=%s",
        (bk_member_id or customer_name, console_id)
    )
```

### Timeline
| Time (MMT) | Event |
|------------|-------|
| 12:38 | C-05 Free → Reserved (booking #961 for 13:45) |
| 13:45 | C-05 Reserved → Active (R gyi, 120 min, ends 15:45) |
| **15:18** | **Booking #975 created** (Sai Aung Hein Ko, 16:00-18:00) → **Active overwritten to Reserved** |
| 15:35 (now) | C-05 still Reserved, booking #961 still Active in DB — status drift |

### Fix Plan
Replace direct `UPDATE console_status SET status='Reserved'` with `_sync_console_status(console_id)` which properly checks for Active bookings first:
1. Active booking found → keep Active ✅
2. No Active + confirmed within 60 min → Reserved ✅
3. Neither → Free ✅

### Affected Code
- `/root/psvibe_api_server/app.py`, line ~2845 (`POST /api/bookings`, customer bot booking creation)
- Same pattern may exist in staff booking creation path — need to audit

## 8. Booking #974 Disappearance — Root Cause & Fixes ✅ (17:00 MMT)

### Investigation Result
Staff Booking flow on C-03 → `step_book_checkin_bind` called **`POST /api/update_booking`** (endpoint doesn't exist → HTTP 405). But `_api_call_async` catches HTTPError and returns dict instead of raising → code thought checkin succeeded (logged success) but booking was NOT updated to Active. Later `_sync_console_status` ran, found no Active booking → set C-03 Free → booking cancelled.

### Fix 1: Dead API Endpoint (HTTP 405)
- **File:** `/root/psvibe-sales-bot/bot/handlers/booking.py` line 1213
- **Before:** `_psvibe_post_async("update_booking", {"id": bk_id, "status": "Active", "console_id": cid})`
- **After:** `_psvibe_patch_async(f"bookings/{bk_id}/status", {"status": "Active", "consoleId": cid})`
- Service: `psvibe-sale-bot` restarted → PID 2294683

### Fix 2: _sync_console_status — NOW() (UTC) → now_mmt()
- **File:** `/root/psvibe_api_server/app.py` lines 96-158
- **Before:** SQL used `NOW()` (MySQL server UTC) to compare against `start_time` (stored as MMT) → ~6.5hr mismatch
- **After:** Generate `_now_mmt_str` and `_now_plus_60` via Python `now_mmt()`, use parameterized `%s`
- Also fixed `DATE_ADD(NOW(), INTERVAL 60 MINUTE)` and `end_time >= NOW()` clauses

### Fix 3: Booking → Active: start_time NOW() → MMT
- **File:** `/root/psvibe_api_server/app.py` line 1623
- **Before:** `start_time=NOW()` in `PATCH /api/bookings/{id}/status` when status→Active
- **After:** `start_time=%s` with `now_mmt().strftime('%Y-%m-%d %H:%M:%S')`
- Service: `psvibe-api` restarted → PID 2294664

### New Lessons
- **#29: HTTP error handling asymmetry** — `_api_call_async` returns error dict for 4xx (no exception) but `_psvibe_post_async` catches Exception → callers never know API failed. Check if response has `status_code` key.
- **#30: MySQL NOW() is UTC** — All time comparisons in SQL must use app-level MMT time passed as parameter. Direct `NOW()` in queries against MMT-stored datetimes will ALWAYS be wrong.
- **#31: Minified Vue SPA fixes must cover BOTH save logic AND template render** — Removing a `v()` condition from save function (`I()`) is not enough; the template ternary `v() ? ... : y("",!0)` also gates the INPUT fields. Miss this = fields invisible, save sends nothing.

### Fix 4-5: Dashboard Edit Booking — Template v() Removal
- **File:** `/root/psvibe_api_server/dashboard-dist/assets/BookingsManagement-BbqxMmPA.js`
- **Fix 4:** `,if(o.value.booking_date...` → `,o.value.booking_date...` (stray `if` after removing `v()){` wrapper)
- **Fix 5:** `v()?(d(),i(T,{key:0},[...time/duration fields...],64)):y("",!0)` → `(d(),i(T,{key:0},[...time/duration fields...],64))` (template always renders fields)

### Charges Separated from Payment
- `paid_amount` = SUM(amount) only (NOT including charges)
- Transfer charges are bank fees, not payment toward supplier/customer
- Account balance deduction = amount + charges (both leave account)
- PNL cost = mmk + charges (effective cost includes fees)
- `buy_charges` and `sell_charges` fields added to PNL API response

### PNL — FIFO Implementation
- **Before:** Average buy rate × realized THB → inaccurate
- **After:** Full FIFO (First In, First Out) matching
- Buy lots ordered by tx_date ASC, created_at ASC, id ASC
- Each sell matched against oldest available buy lot
- Effective rates used: buy = (mmk + charges)/THB, sell = (mmk - charges)/THB
- Both buy and sell charges included in effective rates
- FIFO order preserved via `created_at` timestamp

### PNL Result (June 2026, current data)
- Buy#1: 10,000B @ 125.00 + 300 chg = 125.03/B (FIRST buy)
- Buy#2: 20,000B @ 122.00 + 0 chg = 122.00/B (SECOND buy)
- Sell: 20,000B @ 130.00 - 200 chg = 129.99/B
- FIFO: 10,000 from Buy#1 @ 125.03 + 10,000 from Buy#2 @ 122.00
- PNL: (129.99-125.03)×10,000 + (129.99-122.00)×10,000 = 49,600 + 79,900 = **129,500 Ks**

### Key Lessons
32. **FIFO requires correct chronological order** — use `ORDER BY tx_date ASC, created_at ASC, id ASC`
33. **Charges are bank fees, not supplier payments** — exclude from `paid_amount` but include in PNL
34. **Never include cleanup in test scripts** — caused 2 accidental data deletions
35. **Save-and-restore dropdown selections** when rebuilding options via innerHTML
