after exhaustive debugging across 6+ hypotheses (WAL corruption, bcrypt hash compatibility, path prefix asymmetry, personal project ownership, rate limiting), the **ACTUAL root cause** was a WRONG env var name:

**n8n v2.28.7 reads `this.globalConfig.proxy_hops` from the env var `N8N_PROXY_HOPS`, NOT `N8N_TRUST_PROXY`.**

The env var `N8N_TRUST_PROXY=1` was COMPLETELY IGNORED. Express's `trust proxy` was never set (remained default `false`). Every request through Cloudflare had `X-Forwarded-For` header, but since trust proxy = false, express-rate-limit threw `ERR_ERL_UNEXPECTED_X_FORWARDED_FOR` and BLOCKED the login request.

Curl tests from localhost (127.0.0.1) worked because they didn't have X-Forwarded-For headers — bypassing the issue entirely. This made every fix seem to work from curl but fail from the browser.

**Fix**: Change `N8N_TRUST_PROXY=1` → `N8N_PROXY_HOPS=1` in docker-compose.yml.

## Late Session: bcrypt Hash Compatibility + Login Redirect Debug (18:30–18:49 UTC)

### bcrypt Hash Compatibility Fix — FINAL Layer
Despite the PROXY_HOPS fix, Osmo's browser still showed login failure. Event log showed:
```
n8n.audit.user.login.failed — reason: "wrong credentials"
```

This meant the request reached n8n correctly (no rate limit block), but the password hash didn't match.

**Root cause discovered**: The bcrypt hash in the database was generated by a **DIFFERENT bcrypt implementation** than n8n's runtime bcryptjs.

- `docker run node:20-alpine bash -c "..."` generates a hash using **npm's bcryptjs@2.4.3** installed in the temp container
- n8n container already has **bcryptjs@2.4.3** from its own node_modules
- BUT `docker run` uses a completely SEPARATE Node.js process with its own module resolution
- The hash generated looked valid (`$2a$...` hash), but bcryptjs verification was failing

**Fix**: Generate the hash DIRECTLY inside the running n8n container:
```bash
docker exec n8n node -e "
const bcrypt = require('/usr/local/lib/node_modules/n8n/node_modules/bcryptjs');
console.log(bcrypt.hashSync('osmo123', 10));
"
```
This produced a fresh `$2a$` hash using n8n's own bcryptjs instance. Updated in SQLite:
```sql
UPDATE user SET password = '<hash>' WHERE email = 'osmo@psvibe.com';
DELETE FROM user WHERE email = 'admin@psvibe.com';
-- Re-created admin with fresh hash
UPDATE user SET password = '<hash>' WHERE email = 'admin@psvibe.com';
```

**Lesson #202**: Even when generating with the same *version* of bcryptjs, the hash may be incompatible if generated from a DIFFERENT Node.js process/environment. Always generate inside the target container using `docker exec`.

## Late Session: bcrypt Hash Compatibility + Login Redirect Debug (18:30–18:49 UTC)

### bcrypt Hash Compatibility Fix — FINAL Layer
Despite the PROXY_HOPS fix, Osmo's browser still showed login failure. Event log showed:
```
n8n.audit.user.login.failed — reason: "wrong credentials"
```

This meant the request reached n8n correctly (no rate limit block), but the password hash didn't match.

**Root cause discovered**: The bcrypt hash in the database was generated by a **DIFFERENT bcrypt implementation** than n8n's runtime bcryptjs.

- `docker run node:20-alpine bash -c "..."` generates a hash using **npm's bcryptjs@2.4.3** installed in the temp container
- n8n container already has **bcryptjs@2.4.3** from its own node_modules
- BUT `docker run` uses a completely SEPARATE Node.js process with its own module resolution
- The hash generated looked valid (`$2a$...` hash), but bcryptjs verification was failing

**Fix**: Generate the hash DIRECTLY inside the running n8n container:
```bash
docker exec n8n node -e "
const bcrypt = require('/usr/local/lib/node_modules/n8n/node_modules/bcryptjs');
console.log(bcrypt.hashSync('osmo123', 10));
"
```
This produced a fresh `$2a$` hash using n8n's own bcryptjs instance. Updated in SQLite:
```sql
UPDATE user SET password = '<hash>' WHERE email = 'osmo@psvibe.com';
DELETE FROM user WHERE email = 'admin@psvibe.com';
-- Re-created admin with fresh hash
UPDATE user SET password = '<hash>' WHERE email = 'admin@psvibe.com';
```

**Lesson #202**: Even when generating with the same *version* of bcryptjs, the hash may be incompatible if generated from a DIFFERENT Node.js process/environment. Always generate inside the target container using `docker exec`.
