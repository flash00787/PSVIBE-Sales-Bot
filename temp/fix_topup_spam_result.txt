## Top Up Flow Spam + Stuck Bug Fix - COMPLETED ✅

### Problem Analysis
The Top Up payment flow had a critical message spam issue:

1. **Root Cause:** When user selected a payment method (Cash/KPay/etc), `step_tu_kpay()` would:
   - Send a "Enter amount for [method]" message via `update.message.reply_text()`
   - Return `TU_KPAY` state
   - When user entered amount, it would call `prompt_tu_kpay()` 
   - `prompt_tu_kpay()` would send ANOTHER message with payment status
   - This created **duplicate/spam messages** and confused the flow

2. **Stuck Issue:** The duplicate messages were interfering with the ConversationHandler state transitions, causing users to get stuck and unable to proceed to confirmation.

### Solution Implemented
Applied a **consolidated message flow** approach:

**Before (BUGGY):**
```
User selects "KPay" → step_tu_kpay sends "Enter amount" message → Returns TU_KPAY
User enters "5000" → step_tu_kpay processes → Calls prompt_tu_kpay → Sends ANOTHER message
```

**After (FIXED):**
```  
User selects "KPay" → step_tu_kpay sets flag → Calls prompt_tu_kpay → Single unified message
User enters "5000" → step_tu_kpay processes → Clears flags → Calls prompt_tu_kpay → Updates display
```

### Code Changes Made

1. **Removed duplicate message in `step_tu_kpay`:**
   - When payment method selected, no longer sends separate `reply_text()`
   - Instead sets `tu_show_amount_input = True` flag
   - Immediately calls `prompt_tu_kpay()` for unified display

2. **Enhanced `prompt_tu_kpay` with amount input handling:**
   - Added check for `tu_show_amount_input` flag
   - When flag set, shows amount input prompt for selected method
   - All messages now go through single `prompt_tu_kpay()` function

3. **Added state cleanup:**
   - After amount processed, clears `tu_current_pay_method` and `tu_show_amount_input` flags
   - Prevents state leakage between transactions
