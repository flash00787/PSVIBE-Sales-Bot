# A5 — Write Path: Code → GSheets + DB

## Executive Summary

**TL;DR: The SQLite DB is a READ-ONLY mirror synced from Google Sheets via a separate batch script (`sqlite/setup.py`). The Telegram bot handlers write ONLY to Google Sheets. There is no dual-write code path. All writes are fire-and-forget via `asyncio.create_task()` with receipt confirmation sent BEFORE the write — if the write fails, the user already sees "success" but data is silently lost.**

## Database Write Path — READ-ONLY Mirror

The SQLite database at `/root/Sales-Tele-Bot_refactored/psvibe.db` is a **strictly read-only cache mirror**. It is:

1. **Not imported by ANY bot handler** — `grep -rn "sqlite\|db_manager\|PSVibeDB" bot/` returns zero results
2. **Written ONLY by a batch sync script** at `sqlite/setup.py` which does `DELETE FROM` + `INSERT OR REPLACE ...` for each table
3. **Stands outside the bot entirely** — running it requires a separate Python invocation with `--db` flag

```python
# sqlite/db_manager.py — ONLY methods that WRITE to DB:
def log_sync(...)          # sync_log — records import ops
def import_members(...)    # DELETE FROM members + INSERT OR REPLACE
def import_bookings(...)   # DELETE FROM bookings + INSERT OR REPLACE
def import_sales(...)      # DELETE FROM sales + INSERT OR REPLACE
def import_settings(...)   # INSERT OR REPLACE settings
def import_consoles(...)   # DELETE FROM consoles + INSERT OR REPLACE
def import_console_games(...)
def import_game_library(...)
def import_staff(...)
```

**Direction: Sheets → DB only.** Bot → DB never happens.

## Each Flow Detailed

### Flow 1: New Member — First Purchase
- **Handlers**: `members.py` step_nm_confirm() → _nm_bg()
- **Sheet writes** (via `asyncio.create_task(asyncio.to_thread(_do))`):
  1. `member_sh.batch_update()` — Card_Wallet cols A-D (row_no/id/name/phone), K (staff), M (email)
  2. `topup_sh.batch_update()` — TopUp_Log cols A-C (date, id, tier), E-I (amount, kpay, cash, mins, type), J (staff)
  3. `update_member_effective_rate(nm_id, initial_rate)` — Card_Wallet col L (not for gifts)
  4. (Optional referral) `topup_sh.batch_update()` — extra row for referrer +30 mins
- **ORDER**: Receipt sent to user, THEN all sheet writes fire-and-forget in background
- **ERROR HANDLING**: `try/except` around `asyncio.to_thread(_do)` → only `logging.error`. User sees success regardless.

### Flow 2: Top Up
- **Handlers**: `members.py` step_tu_confirm() → _tu_bg()
- **Sheet writes**:
  1. `topup_sh.batch_update()` — TopUp_Log cols A-C, E-I (single row)
  2. `update_member_effective_rate(tu_id, new_rate)` — Card_Wallet col L (weighted avg)
- **ORDER**: Receipt first, then background write
- **ERROR HANDLING**: `try/except` → `logging.error`. No user notification.
- **Post-top-up flow**: If from `console_sale` shortfall, restores session snapshot and continues sale flow.

### Flow 3: Daily Sales
- **Handlers**: `sales.py` step_sale_confirm() → _sale_bg()
- **Sheet writes** (all background `asyncio.create_task`):
  1. `sales_sh.batch_update()` — Sales_Daily cols A-K (date/voucher/member/console/mins/game/food/discount/net/kpay/cash), N (wallet_deduct_mins), O (staff)
  2. `stock_sh.append_row()` — Stock_Out_Log per food item (date, voucher, item, qty, unit_price, subtotal, cost_price, cogs)
  3. `_update_inv_total_k1()` — Inventory sheet K1 cell
  4. (Optional) `_replit_post("sheets/promotions-log")` — Promotions log via REST
  5. (Optional) `member_sh.update_cell()` — Bonus mins to Card_Wallet col H
  6. `_replit_patch(f"bookings/{bk_id}/status")` — Mark linked booking as completed
- **Double-confirm guard**: `_sale_saved` flag prevents duplicate save
- **ORDER**: Receipt first, then background writes + booking completion separately
- **ERROR HANDLING**: `try/except` → `logging.error`. User sees success regardless.

### Flow 4: Session End / Booking
- **Handlers**: `__init__.py` end_booking()
- **Sheet writes** (synchronous):
  - Reads all rows, iterates to find matching booking ID
  - `sh.update(f"F{i}", [[now.strftime("%H:%M")]])` — End Time
  - `sh.update(f"G{i}", [["Done"]])` — Status
- **ERROR HANDLING**: `try/except` → returns False

### Flow 5: Create Booking
- **Handlers**: `__init__.py` create_booking()
- **Sheet writes** (synchronous):
  - `sh.append_row()` — Console_Booking (bk_id, date, console, member, start_time, planned_end, Active, staff, notes)
- **ERROR HANDLING**: No try/except in the function (returns bk_id directly)

### Flow 6: Stock Out (Manual)

- **Sheet writes** (synchronous):
  - `stock_sh.append_row()` — Stock_Out_Log

### Flow 7: Stock In (PIN-gated)

### Flow 8: Finance — OPEX

## CRITICAL ISSUES FOUND

### 🔴 CRITICAL: Receipt-Before-Write Anti-Pattern (Members & Sales)
- [ ] **step_nm_confirm()** (members.py:L719-724, L735-771): Receipt sent BEFORE background GSheets write. If API fails, user sees success but data is permanently lost.
- [ ] **step_tu_confirm()** (members.py:L1067-1071, L1079-1089): Same pattern — receipt first, then fire-and-forget write.
- [ ] **step_sale_confirm()** (sales.py:L907-931, L971-1030): Same pattern with EVEN MORE background writes (sales + stock + promotions + bonus + booking). Largest blast radius for silent data loss.

### 🟡 HIGH: Multi-Sheet Writes Without Transaction Support
- [ ] **New Member flow writes 2-4 sheets** (Card_Wallet, TopUp_Log, possibly TopUp_Log again for referral) — no rollback if mid-sequence fails
- [ ] **Daily Sales flow writes up to 5 sheets** (Sales_Daily, Stock_Out_Log per item, Promotions_Log, Card_Wallet col H, Booking status) — no coordination
- [ ] **Stock Out writes 2 sheets** (Stock_Out_Log + Inventory K1) — K1 update can fail after append

### 🟡 HIGH: Sequential Cell Updates Without Atomicity (Asset Dispose)
- [ ] `finance.py:L809-816`: 4 sequential `update_cell()` calls. If call 2 fails, col H shows "Disposed" but cols I-K have stale values.

### 🟡 HIGH: Payable/Receivable Settle — Partial Update Risk
- [ ] `finance.py:L1935-1938`: 2-3 sequential `update_cell()` calls. Col F could change to "Paid" while col G (date) remains stale.

### 🟡 MEDIUM: Read-Modify-Write Race Condition (Asset Dispose Proceeds)
- [ ] `finance.py:L811-812`: Read col K value then write col K = old + new. Two concurrent disposals on same asset lose data.

### 🟡 MEDIUM: Read-Modify-Write Race Condition (Bonus Mins)
- [ ] `sales.py:L1014-1018`: Read col H (wallet balance) then write col H (new balance). Two concurrent bonus-awarding sales on same member lose increment.

### 🟡 MEDIUM: Sync vs Background Write Inconsistency
- [ ] Session end (`end_booking`) is synchronous but the corresponding sales entry is background. `end_booking` can succeed while the sale write silently fails.

### 🟡 MEDIUM: Effective Rate Write Target May Be Stale
- [ ] `members.py:L1737-1746`: `update_member_effective_rate()` reads all rows fresh, then writes one cell. If background new-member write hasn't completed, the row index may be wrong.

### 🟢 LOW: Stock In — Loop Writes Without Partial Rollback
- [ ] `stock_in.py:L246`: For-each-item append. If item 3/5 fails, items 1-2 are committed with no rollback.

### 🟢 LOW: No Write Retries
- [ ] All write paths have single-attempt with no retry logic. Google Sheets API rate limits or transient failures become permanent data loss.

### 🟢 LOW: DB Mirror Lag
- [ ] The SQLite DB is a batch-synced snapshot. Dashboards using DB reads show stale data until the next scheduled sync.
