# A12 — API Rate Limits & Batch Operations

## Executive Summary

The bot has a **solid retry foundation** (`_sheets_retry` decorator with exponential backoff for 429/500/503) but **batch operations are underutilized** and **there is no rate-limit quota tracking**. The three busiest write flows (New Member, Top Up, Daily Sales) each issue **4-8+ API calls** where 2-3 calls could be sufficient with proper batching. A sale with 5 food items currently triggers **7 API calls** to Google Sheets -- at peak hours with multiple staff, the 60 req/min/user quota could be hit.

---

## 1. Retry/Resilience Infrastructure

### Present: `_sheets_retry` Decorator

**Location:** `bot/__init__.py` L41-86

```python
_SHEETS_RETRY_CODES = (429, 500, 503)
_SHEETS_MAX_RETRIES = 3
_SHEETS_BASE_DELAY  = 1  # seconds
```

**What it does:**
- Wraps 18 gspread.Worksheet methods (`update_cell`, `append_row`, `batch_update`, `get_all_values`, `col_values`, etc.) via monkey-patching at module load.
- On 429/500/503: retries up to 3 times with exponential backoff (1s, 2s, 4s = **7s max total delay**).
- On 403: **never retries** -- raises `SheetsPermissionError` with critical log detailing which Service Account and Sheet ID are affected.
- On connection/timeout/OSError: same retry logic as 429.

**Verdict:** Good foundation. Covers the most common transient failures.

### Gap: No Request/minute Quota Awareness

Google Sheets API quotas:

| Quota | Limit |
|-------|-------|
| Read requests per 100 seconds per user | 300 |
| Write requests per 100 seconds per user | 300 |
| **Read requests per minute per user** | **60** |
| **Write requests per minute per user** | **60** |
| Requests per 100 seconds per project | 3,000 |

The bot **does not track or rate-limit its own request rate**. If a burst of operations occurs (e.g., 3 staff each saving a sale simultaneously, with 5 food items each = ~21 API calls in seconds), the 60 req/min/user threshold could be reached.

### Gap: Write Paths Bypass Further Retries

The `_sheets_retry` decorator only wraps gspread methods at the module level. But the three busiest flows (New Member, Top Up, Daily Sales) have their own `try/except` that catches everything:

```python
try:
    await asyncio.to_thread(_do)
except Exception as _e:
    logging.error("nm_bg_write: %s", _e)
```

This catch-all after `_sheets_retry` exhausted its 3 retries would log but never retry further. The real problem is that the receipt was sent BEFORE the write, so the user has **no way to know** the data was lost.

### Present: Customer Bot API Retry Layer

**Location:** `customer_bot/api.py` L84-120

The customer bot's HTTP API client has its own separate retry system:
- `_RETRY_BASE_DELAY = 1`, `_MAX_RETRIES = 4`
- Exponential backoff on all 5xx/network errors
- Handles 429 with retry-after headers

### Present: Broadcaster Rate Limit Awareness

**Location:** `handlers/broadcast.py` L61

```python
await asyncio.sleep(0.05)   # ~20 msg/sec - within Telegram rate limits
```

The broadcast handler is aware of Telegram's rate limits (not Sheets, but good practice). **No equivalent awareness for Sheets.**

## 4. Rate Limit Handling Assessment

### Current State: REACTIVE ONLY

The bot has **no proactive rate limiting**. It only reacts when Google returns a 429.

| Aspect | Status | Details |
|--------|--------|---------|
| Quota tracking (req/min) | NONE | No token bucket, no counter |
| Pre-429 throttling | NONE | No delay between API calls |
| Burst detection | NONE | No awareness of concurrent writes |
| Retry on 429 | YES | 3 retries, 1s/2s/4s backoff |
| Retry on 500/503 | YES | Same exponential backoff |
| Retry on connection errors | YES | Same exponential backoff |
| Critical error (403) handling | YES | Immediate halt + critical log |
| Write failure user notification | NONE (3 flows) | Silent data loss for NM/TU/Sales |
| Circuit breaker | NONE | Keeps hitting API indefinitely |
| Jitter in retry delays | NONE | All instances retry at same offset |

### Rate Limit Risk Assessment

**Scenario calculation:** Peak hour with 3 staff members:

| Operation | API Calls each | x 3 staff | Total calls |
|-----------|---------------|-----------|-------------|
| 3 x Daily Sales (avg 3 food items) | 8 | x 3 | 24 |
| 2 x Top Ups | 4 | x 2 | 8 |
| 1 x New Member | 5 | x 1 | 5 |
| 1 x Stock Out | 2 | x 1 | 2 |
| 1 x End Booking | 1 | x 1 | 1 |
| 2 x Create Booking | 1 | x 2 | 2 |
| **Total in ~2-minute burst** | | | **42 API calls** |

At 42 calls in 2 minutes (21 req/min average), this is **well under the 60 req/min/user hard limit**. However:

- **Burst spikes could exceed the limit momentarily** (all writes happen in background threads simultaneously)
- With `_sheets_retry` only allowing 3 retries (7 seconds total), a rate-limited call during a real burst could fail permanently
- **No jitter** means all concurrent retries hit Google at the same time

**Risk Level: MEDIUM** -- unlikely under normal load but no safety margin for bursts.

## 5. Critical Bottlenecks Identified

### RED Bottleneck #1: Per-Item Stock Out Writes (Sales Flow)
**Impact:** A sale with 5 food items makes 5 separate `append_row()` calls instead of 1 `batch_update`.
**Waste:** 4 extra API calls per sale with food items.
**Fix:** Collect all food items into a single `batch_update` call with 5 ranges.

### RED Bottleneck #2: Missing `_update_inv_total_k1` Function
**Impact:** Called in `sales.py:L993`, `stock.py:L184`, `stock_in.py:L252` but **NOT DEFINED** anywhere in the codebase. This would cause a `NameError` at runtime, silently caught by the outer `try/except` and preventing inventory total updates.
**Fix:** Define the function or verify it exists in a file not on disk.

### RED Bottleneck #3: Sequential Cell Updates Without Atomicity
**Impact:** Asset Disposal writes 4 cells one at a time. If Google Sheets fails after cell 1, the asset shows "Disposed" but date/qty/proceeds are stale. Settle has the same problem (2-3 cells).
**Fix:** Use a single `batch_update` call with all ranges for the same row.

### YELLOW Bottleneck #4: Effective Rate - 2 Calls Where 1 Would Do
**Impact:** Every Top Up and New Member calls `get_all_values()` to find the member row, then `update_cell()` to write. The read-write cycle is unnecessary -- the row number is already known from the `next_write_row` call.
**Fix:** Pass the row number through instead of re-reading the entire sheet.

### YELLOW Bottleneck #5: No Write-Ahead Acknowledgement (Data Loss Risk)
**Impact:** Receipt sent to user -> background write fails -> user sees "success" but data is gone. Affects the 3 highest-value flows (New Member, Top Up, Daily Sales).
**Fix:** Either write first then send receipt, or store write queue with retry.

### YELLOW Bottleneck #6: Refresh Calls Bypass Caching
`sales.py:L994`: `_replit_get("sheets/inventory?nocache=1")` -- sends a cache-busting HTTP call after every sale, hitting the API server which likely hits Google Sheets again.
**Impact:** Doubles the inventory read load unnecessarily after each sale.

## 6. Optimization Recommendations

### Priority 1: Batch Stock Out Items (HIGH IMPACT, LOW EFFORT)

**Before (sale with 5 food items):**
```python
for item in food_sold:           # 5 separate append_row calls
    stock_sh.append_row([...])
```

**After:**
```python
ranges = []
for item in food_sold:
    ranges.append({
        "range": f"A{so_row}:H{so_row}",
        "values": [[today, v_no, item["name"], item["qty"], ...]]
    })
    so_row += 1
stock_sh.batch_update(ranges, value_input_option="USER_ENTERED")  # 1 call
```

**Expected gain:** 5 calls -> 1 call for typical sale. **80% reduction in Stock Out writes.**

### Priority 2: Batch Finance Sequential Writes

**Before (Asset Dispose):**
```python
sh.update_cell(sheet_row, 8, "Disposed")   # call 1
sh.update_cell(sheet_row, 9, date)          # call 2
sh.update_cell(sheet_row, 10, qty)          # call 3
sh.update_cell(sheet_row, 11, proceeds)     # call 4
```

**After:**
```python
sh.batch_update([
    {"range": f"H{sheet_row}:K{sheet_row}",
     "values": [["Disposed", date, qty, proceeds]]}
], value_input_option="USER_ENTERED")
```

**Expected gain:** 4 calls -> 1 call. **75% reduction for Asset Dispose, 50% for Settle.**

### Priority 3: Pass Row Numbers for Effective Rate

**Before:**
```python
def update_member_effective_rate(member_id, new_rate):
    rows = member_sh.get_all_values()              # reads ALL members
    for i, row in enumerate(rows[1:], start=2):
        if row[1] == member_id:
            member_sh.update_cell(i, 12, rate)      # writes one cell
```

**After:**
```python
def update_member_effective_rate(row_num, new_rate):
    member_sh.update_cell(row_num, 12, rate)       # single write, no read
```

The caller already has `cw_row` from `next_write_row(member_sh)` for new members, or knows the row from Top Up flow.

**Expected gain:** 2 calls -> 1 call. **50% reduction for effective rate writes.**

### Priority 4: Remove Unnecessary Cache Busting

`sales.py:L994`: Remove `_replit_get("sheets/inventory?nocache=1")` -- the `_update_inv_total_k1()` already invalidates what's needed.

**Expected gain:** 1 HTTP call removed per sale with food.

### Priority 5: Add Jitter to Retry Delays

**Before:**
```python
delay = _SHEETS_BASE_DELAY * (2 ** attempt)   # 1s, 2s, 4s
```

**After:**
```python
delay = _SHEETS_BASE_DELAY * (2 ** attempt) + random.uniform(0, 0.5)
```

**Expected gain:** Prevents thundering herd when multiple threads hit 429 simultaneously.

### Priority 6: Proactive Rate Limiting

Add a simple token bucket (10 tokens/sec, burst 30) before gspread calls:

```python
class RateLimiter:
    def __init__(self, rate=10, burst=30):
        self.tokens = burst
        self.max_tokens = burst
        self.rate = rate
        self.last_fill = time.monotonic()
        self._lock = threading.Lock()

    def acquire(self):
        with self._lock:
            now = time.monotonic()
            elapsed = now - self.last_fill
            self.tokens = min(self.max_tokens, self.tokens + elapsed * self.rate)
            self.last_fill = now
            if self.tokens >= 1:
                self.tokens -= 1
                return
        time.sleep((1 - self.tokens) / self.rate)

_sheets_limiter = RateLimiter(rate=10, burst=30)
```

**Expected gain:** Prevents ever hitting the 60 req/min/user hard limit. **Zero 429s.**
